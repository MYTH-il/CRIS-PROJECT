Data strategy — tackle this first

Recommended
Approach A — RTI + public judgment corpora
Low data needed
File RTI requests for anonymized FIR summaries and pair them with Indian Kanoon + JUDIS judgments (public, FIR-like narrative style). Use NCRB tables for taxonomy grounding. This becomes your real-data backbone.
Pros

Legally obtained. Real distribution. FIR-style language. Highly defensible for research.

Cons

RTI process is slow (30–60 days). Data may be redacted. Manual cleaning required.

Approach B — Rebuild synthetic generator (IPC-grounded)
Immediate
Generate FIRs that reference real IPC sections, real Indian cities/districts, and genuine FIR linguistic patterns. This is the fastest way to dramatically raise synthetic realism.
Pros

Immediate. IPC-aligned. Realistic narratives. No legal/privacy risk.

Cons

Still synthetic — cannot support real-world accuracy claims alone.

Approach C — Zero-shot with IndicBERT / XLM-R
Research angle
Use Hugging Face Inference API for zero-shot classification/NER. This bypasses WDAC (no local DLLs) and gives a strong research baseline while real data is pending.
Pros

Works with zero labeled Indian data. Strong on multilingual. WDAC-safe.

Cons

Latency + API cost. Lower ceiling than fine-tuned. On-prem is harder.

Approach D — Active learning loop
Long-term
Deploy the system with human-in-the-loop annotation. As officers interact, confident model predictions are auto-labeled; uncertain ones get flagged for expert review. Model improves continuously with minimal labeled data.
Pros

Self-improving. Adapts to local patterns. Data grows with usage. Most production-realistic.

Cons

Needs a live deployment to work. Cold start problem. Requires annotation UI + review workflow.

Stack all four approaches. Start with B now to fix synthetic realism. File RTI requests and collect judgments (A) in parallel. Use C (HF Inference API) as your strong baseline while real data arrives. Add D once a prototype is stable.

Rust Target (low-risk, high-impact): build a small ingestion + validation CLI that reads CSV/JSON, enforces schema, and outputs clean parquet/CSV for Python. This keeps the system cohesive: Rust only handles I/O and validation, Python stays the orchestration layer.
Real data acquisition
  • File RTI requests — target State Crime Record Bureaus (SCRB) for anonymized FIR data. Aim for UP, Maharashtra, Tamil Nadu (largest datasets). Use RTI Online portal.
  • NCRB open data — download and parse all publicly available NCRB crime statistics tables (2018–2023). They have district-level breakdowns by IPC section.
  • Court judgment corpora — Indian Kanoon and JUDIS have machine-readable judgments. These contain FIR-like narratives and are publicly accessible.
LLM-augmented synthetic upgrade
  • Rebuild generate_crime_dataset.py — rewrite to use GPT-4/Claude API. Ground each generated FIR in a real IPC section, real Indian city/district, real crime category from NCRB taxonomy. Add Hindi field variants.
  • Schema validation layer — add a Pydantic model or pandera schema that enforces column types, lat/lon bounds for India, valid IPC section codes, date ranges. Fail loudly at load time.
Phase 2 NLP pipeline upgrade Weeks 2–5 ▶
Classification
  • Add Logistic Regression baseline — fast, trains on all 60k rows in seconds, gives honest metrics. This is your research baseline. MLP is the improved model. Compare both properly.
  • Zero-shot with IndicBERT or XLM-R — use Hugging Face Inference API (no local install, no WDAC problem) to call a multilingual model in zero-shot mode. This becomes your "strong model" comparison in the paper.
  • Proper train/val/test split — 70/15/15. Fixed seed. Report F1, precision, recall, confusion matrix. Never touch the test set during development.
NER upgrade (WDAC-safe)
  • Rule-based + regex NER — build a custom extractor for: IPC sections (regex), FIR numbers, Indian phone numbers, vehicle plates (state code + format), police station names (lookup table), weapon keywords (curated list).
  • Hugging Face Inference API for NER — call a hosted token-classification model for PERSON, ORG, LOC. No local install. Falls back to regex if API is unavailable.
Severity scoring improvement
  • IPC section severity map — create a lookup table mapping IPC sections to severity tiers (cognizable/non-cognizable, bailable/non-bailable, punishment quantum). This is domain-accurate and requires no ML.
Phase 3 Case linking & semantic search Weeks 4–6 ▶
  • Sentence-transformers via API — replace TF-IDF cosine similarity with embedding-based similarity using Hugging Face Inference API (paraphrase-multilingual-MiniLM). Same WDAC-safe pattern.
  • ChromaDB or FAISS for vector storage — ChromaDB is pure Python (WDAC-safe). Pre-compute embeddings for all 60k reports, store in ChromaDB, query at runtime. This makes MO matching actually semantic.
  • Case clustering with BERTopic — use BERTopic (API-based topic extraction) to cluster similar but uncategorized crimes. Surface clusters in the UI as "Emerging Pattern Alerts."
  • Cross-case entity linking — if the same suspect name or vehicle plate appears in multiple cases, link them. Build a basic coreference graph using entity string matching + fuzzy match (rapidfuzz — pure Python).
Phase 4 Geospatial & predictive improvements Weeks 5–7 ▶
  • Switch to H3 spatial binning — replace coordinate rounding with Uber's H3 library (pure Python wheels available). Consistent hexagonal grid, more stable hotspot boundaries, better at Indian geography.
  • Add Prophet/ARIMA as temporal baselines — Prophet is pure Python (Facebook). Run it alongside RandomForest. Show both in the UI. Lets you quantify what the spatial dimension adds. Required for research paper comparison table.
  • India-specific geodata — integrate district/taluka boundaries from the Survey of India open data. Replace generic lat/lon with district-level aggregations. Map crimes to police station jurisdictions.
  • Clearly label predictions as probabilistic — add UI copy: "This is a statistical likelihood map based on historical patterns, not a guarantee. For resource allocation guidance only." Required for ethics compliance.
Phase 5 Backend, API & production architecture Weeks 7–10 ▶
  • FastAPI backend — decouple ML logic from Streamlit. Build REST endpoints: /classify, /extract-entities, /similar-cases, /forecast. Streamlit becomes a thin frontend client.
  • PostgreSQL for structured data — replace CSV loading with a proper DB. Cases, entities, predictions, audit log all in separate tables. Use SQLAlchemy ORM.
  • ChromaDB / pgvector for embeddings — production vector store with persistent storage, indexing, and filtered search (e.g. "find similar robbery cases in Maharashtra only").
  • Authentication & access control — JWT-based auth. Role-based access: Investigator, Analyst, Admin. All queries logged with user ID for audit trail.
  • Docker + CI/CD — Dockerize the FastAPI + Streamlit stack. GitHub Actions pipeline: lint → test → build image → deploy.
  • Frontend transition plan — keep Streamlit as the interim UI for research demos. In parallel, build the production frontend (React or similar) against the same API. This avoids blocking model work on UI.
  • Rust accelerators (targeted) — add small Rust services only where they pay off: ingestion/validation, embedding search microservice, or API gateway. Keep Python as the main orchestration layer.
Phase 6 Ethics, governance & evaluation Weeks 8–11 ▶
  • Bias audit — evaluate model performance broken down by religion, caste indicators, and geography. If accuracy varies significantly across groups, document it and apply calibration. Required for law enforcement applications.
  • PII anonymization pipeline — strip or mask: names, Aadhaar numbers, phone numbers, addresses before data enters any model. Use presidio (Microsoft, pure Python) or a custom regex masker.
  • No-arrest-prediction hard block — add a UI-level and API-level guardrail that prevents the system from being used to predict who will commit a crime. Only patterns and resource allocation.
  • Research evaluation pipeline — implement proper held-out test set evaluation. Report F1 by class, confusion matrix, NER span-level F1, hotspot prediction MAE. Separate synthetic vs real results clearly.
  • Explainability layer — add LIME or SHAP (both scikit-compatible) to explain why the model classified a report as a specific crime type. Required for investigator trust and legal defensibility.

  • Hindi + regional language support — add IndicBERT or AI4Bharat's IndicNLP pipeline. Indian FIRs are often written in Hindi, Marathi, Tamil. The system must handle transliteration and code-mixing.
  • Paper structure — Introduction → Related Work (NLP in Indian law enforcement) → Dataset (synthetic + RTI, clearly separated) → System Architecture → Experiments (model comparisons, ablation study) → Ethics → Conclusion.
